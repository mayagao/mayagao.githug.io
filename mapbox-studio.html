A Powerful New Mapmaking Tool Fit for Both Pros and Newbies
I started working on Mapbox Studio four months before its public launch. A primary focus of the initial release was the Mapbox Studio style editor, a professional map design tool. As the first visual designer in the team, I was tasked to create a more friendly, consistent and polished visual language system for Studio's editor and dashboard.
Style Editor
Before I joined, the team has spent two years working out the map renderer, GL specification, Styles API, and other technologies that supports the editor. They have done countless iterations on the visual interface to ensure we are building a software that's accessible to more than just professional cartographers.
Powered by the fast, client-side GL renderer, Studio allows users to directly select and edit map layers and view changes instantly.
Style Editor Iteration August, 2015

Ramp UI
Ramp provides designers an easy way to adjust and edit values based on different zoom levels. It was definitely one of the biggest design challenges we have in the form. The main three challenges are:
  1. There are different types of ramps, categorical, color, and number.
  2. Help users understand the change trend through graph is tough. It's hard to know how values will change based on zoom levels.
  3. There are specific rules to make a ramp function valid and it's easy to make mistalkes.
Ramps allows for map design based on zoom levels.
Here are some explorations I designed to solve these problems:
Ramp popover popover
I duplicated the ramp, large, in the popover. Add headings to the fields. Used three different designs to differentiate three types of ramps.
Position indicator
Users don’t understand how values for other zoom levels are going to change. So I added this diamond-shaped pointer to indicate to users values at the current zoom level.

What I have learned
It was definitely quite intimidating and scary at first, as someone very new to the company, to be involved in a product that has a long, complicated development history.
Coming to the team at a later stage definitely makes the job more challenging since I have far less context about the product development history. A big part of what I do everyday at the beginning was just reading. I want to better understand everything from why the team decided to drop certain features to the technology foundations behind certain UI elements. At Mapbox we value deeply about transparent and open communication and we try to document everything on Github, allowing people to understand things without any previous background knowledge. I have to say as someone starting out I really benefited tremendously from that culture.
